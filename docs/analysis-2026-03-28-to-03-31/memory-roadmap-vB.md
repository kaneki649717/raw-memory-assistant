# 记忆系统 5 大问题改造路线图（版本 B）

## 结论
当前记忆系统确实可以按 **5 个核心问题** 来拆解治理。建议按“先统一中枢、再稳定写入、再增强回忆、最后收口主运行时”的顺序推进。

---

## 问题 1：双轨并存，记忆中枢未统一
### 现状
目前并存两条主链路：
- 官方链路：`MEMORY.md / memory/*.md + memory_search + memory_get + SQLite/向量索引`
- 实验链路：`L0/L1/L2 + hybrid recall + replay + 未来 rerank/orchestrator`

### 风险
- 记忆写入目标不统一
- 查询入口不统一
- 维护成本翻倍
- 同一事实可能在多层重复或冲突

### 目标
建立 **统一记忆总线**：
- 定义什么进 `daily memory`
- 定义什么进 `MEMORY.md`
- 定义什么进 `L1 decision`
- 定义什么保留在 `L2 replay`
- 定义查询总入口与召回优先级

### 改造步骤
1. 先明确四类记忆对象：事件、决策、偏好、原话证据
2. 为每类对象定义唯一主落点
3. 统一 recall orchestrator：先 structured，再 durable，再 raw replay
4. 把实验 recall 接到主运行时外层，而不是并行裸跑
5. 明确用户“记住这个”时的落盘策略

### 完成标准
- 所有写入都有唯一归属
- 所有查询走统一 recall 编排
- 不再出现“两套记忆都在回答”的情况

---

## 问题 2：官方持久记忆偏 Markdown，结构强度不够
### 现状
`MEMORY.md` 和 `memory/YYYY-MM-DD.md` 适合人读人改，但对机器推理不够强。

### 风险
- 决策、偏好、事件混写
- 时间、因果、结果字段缺失
- 检索后还需要二次理解

### 目标
让 Markdown 继续保留，但把它从“唯一主结构”升级成“人类可读视图”；真正机器可用的事实层放到结构化 working memory 中。

### 改造步骤
1. 固化 event / decision / preference / todo schema
2. L1 中补齐 why / outcome / status / timestamp / entities
3. Markdown 侧改成摘要镜像，不承担全部结构职责
4. 为长期记忆建立“提升规则”：什么值得从 daily 提升到 MEMORY.md
5. 引入轻量校验，避免坏结构写入主存

### 完成标准
- 机器读结构化层，人读 Markdown 层
- 重要决策能直接按字段提取，不靠猜
- 长期记忆不再只是散文式累积

---

## 问题 3：记忆写入机制偏被动，不够稳定
### 现状
现在 durable memory 主要靠：
- 用户主动要求记住
- agent 临场自觉写入
- compaction 前 memory flush 补写

### 风险
- 容易漏记
- 容易误记
- 写入时机不可预测
- 质量波动大

### 目标
把“记忆写入”从补救动作，升级成稳定流水线。

### 改造步骤
1. 在每轮或每 N 轮后做轻量 ingest 判断
2. 把“是否值得写入”变成规则 + 模型联合决策
3. 将 flush 保留为兜底，不再作为主写入机制
4. 对高价值信号设明确触发器：用户偏好、长期项目状态、踩坑经验、稳定约定
5. 引入低成本质量门控：去重、弱摘要过滤、空洞结论过滤

### 完成标准
- 关键事实不会只靠 compaction 前抢救
- 写入时机稳定
- 记忆污染率下降

---

## 问题 4：会话历史虽完整，但还没真正转化为高质量 episode memory
### 现状
session transcript 很完整，但更多是“聊天档案”，还不是高质量经验记忆。

### 风险
- 只能搜到文本，不能稳定恢复事件链
- 多轮因果关系弱
- 原话有了，但经验抽象不足

### 目标
把 session transcript 从“原始档案”提升为“可回放、可归因、可抽象”的 episode memory 来源。

### 改造步骤
1. 以 turn / exchange / task 为单位抽取事件
2. 建立 transcript -> event -> decision -> replay 的引用关系
3. 让 L2 replay 不只是原文存档，而是带定位与上下文边界
4. 对重复主题形成 episode 聚合，而不是散落多条文本
5. 将 session memory 与 structured memory 共用一套 evidence 引用方式

### 完成标准
- 能回答“当时怎么说的”“为什么这么决定”“后面结果如何”
- 不是只搜到一句相似文本，而是能还原事件片段

---

## 问题 5：实验 working-memory 方向正确，但还没完全收口进主运行时
### 现状
L0/L1/L2、hybrid、replay 已经有雏形，但还没完全接管主链路。

### 风险
- 线上行为与实验行为不一致
- debug 成本高
- 迁移迟迟不收口

### 目标
把实验系统从“旁路增强”变成“正式中枢模块”。

### 改造步骤
1. 先补齐 hybrid 中文 lexical / entity hit
2. 接入 rerank，形成稳定 recall 排序
3. 把 recall bundle 标准化，作为统一输出协议
4. 接主运行时钩子：写入、检索、用户显式记忆命令、session recall
5. 保留回退机制：主链路失败时仍可退回官方 memory_search

### 完成标准
- 主运行时默认走 structured recall
- 官方 memory_search 退居兼容/回退层
- 实验系统不再是“另一个系统”，而是正式系统的一部分

---

## 推荐推进顺序（非常关键）
### 第一优先级
**问题 1：统一记忆中枢**
因为不先统一，后面所有优化都会继续分叉。

### 第二优先级
**问题 3：稳定写入机制**
因为没有稳定写入，再强的 recall 也只是搜残缺数据。

### 第三优先级
**问题 5：把 working-memory 正式接入主运行时**
因为你真正想要的能力，核心都在这一层。

### 第四优先级
**问题 4：把 session transcript 升级成 episode memory**
这是把“原始档案”变成“经验记忆”的关键。

### 第五优先级
**问题 2：优化 Markdown 层定位**
这个要做，但更像结构归位和长期整理，不是最先卡脖子的点。

---

## 执行节奏建议
### 阶段 1：统一口径
- 定义 5 类记忆对象与唯一归属
- 统一 recall 入口
- 统一写入判定规则

### 阶段 2：写入稳定化
- ingest 自动化
- flush 降级为兜底
- 去重/质量门控上线

### 阶段 3：回忆质量提升
- hybrid 优化
- rerank 上线
- replay 证据链完善

### 阶段 4：主运行时收口
- structured memory 接管主链路
- official memory_search 作为回退
- 统一监控与调试入口

## 后续汇报机制（按你的要求）
后续就按 **5 个问题** 来管理。每解决一个问题，就给你手机发一次：
- 这次解决了什么
- 怎么解决的
- 当前还剩下哪几个问题
- 下一步准备解决哪个

这样你手机上会持续收到一条条清晰进展，而不是大而散的技术日志。
